\chapter{ALGORITHM AND MATHEMATICAL MODEL}

To achieve high operational determinism and security, the Nebula platform implements specific algorithms for its most critical subsystems: the AI Troubleshooting Copilot, the resource normalization layer, the task retry scheduler, and the credential security vault. This chapter formalizes those models through pseudo-code, state descriptions, and mathematical notation. The purpose of the chapter is not to claim theoretical novelty in every component, but to show that the control plane can be reasoned about rigorously rather than described only at a high level.

\section{The AI Copilot: Adaptive Log Remediation Algorithm}
The AI Copilot is designed to solve the "semantic mismatch" problem in multi-cloud error logs. When a Terraform task fails, the logs are often lengthy, noisy, and highly provider-specific. The Adaptive Log Remediation (ALR) algorithm reduces this noise by combining deterministic preprocessing with targeted inference.

Let a raw execution log be represented by the token sequence
\[
L = \{l_1, l_2, \ldots, l_n\}
\]
where each token may include provider messages, timestamps, stack traces, resource identifiers, or sensitive values. The goal of the remediation pipeline is to produce a structured tuple
\[
R = (c, s, a, \gamma)
\]
where $c$ is the failure category, $s$ is a concise summary, $a$ is a suggested action, and $\gamma \in [0,1]$ is the confidence score.

The pipeline applies four stages:
\begin{enumerate}
    \item \textbf{Sanitization}: remove tokens that may contain secrets, internal IPs, or organization-specific identifiers.
    \item \textbf{Context enrichment}: attach resource type, provider, region, and task metadata.
    \item \textbf{Classification}: identify likely failure families such as authentication, quota, syntax, dependency, or connectivity.
    \item \textbf{Action mapping}: convert the detected family into a deterministic remediation recommendation where possible.
\end{enumerate}

\subsection{Pseudo-code for the ALR Algorithm}
\begin{lstlisting}[language=Python, caption=AI Copilot Remediation Logic]
def execute_remediation_pipeline(task_id: int, raw_log: str) -> dict:
    # Stage 1: Scrubbing and anonymization
    scrubbed_log = scrub_secrets(raw_log)
    trimmed_log = prioritize_error_windows(scrubbed_log, max_lines=80)

    # Stage 2: Context injection
    ctx = get_task_context(task_id)
    prompt = build_prompt(
        provider=ctx.provider,
        resource_type=ctx.resource_type,
        operation=ctx.operation,
        region=ctx.region,
        log_excerpt=trimmed_log,
    )

    # Stage 3: Controlled inference
    llm_output = llm.query(prompt, temperature=0.1)

    # Stage 4: Deterministic post-processing
    category = classify_failure(llm_output, trimmed_log)
    remedy = map_category_to_remedy(category, ctx)
    confidence = score_confidence(category, llm_output, trimmed_log)

    return {
        "task_id": task_id,
        "category": category,
        "summary": llm_output.get("summary"),
        "remedy": remedy,
        "confidence": confidence,
        "auto_fix": category in {"quota", "missing_tag", "retryable_lock"},
    }
\end{lstlisting}

\subsection{Confidence Scoring Model}
The confidence score should not be a vague heuristic. Nebula models the final confidence value as a weighted combination of three signals:
\[
\gamma = \alpha p + \beta m + \delta h
\]
where $p$ is the classifier probability, $m$ is the pattern match strength against known error signatures, and $h$ is the historical success rate of the suggested remediation for similar failures. The weights satisfy
\[
\alpha + \beta + \delta = 1
\]
with $\alpha > \beta > \delta$ in the default configuration because current log evidence is prioritized over historical similarity.

\section{Unified Resource Normalization Algorithm}
One of the central platform challenges is converting provider-specific metadata into a canonical model that the frontend and database can understand consistently. Let the provider-specific resource response be denoted by $X_p$, where $p \in \{\text{AWS}, \text{Azure}, \text{GCP}\}$. The normalization function
\[
f_p(X_p) = Y
\]
maps this provider response into a canonical tuple
\[
Y = (\text{id}, \text{name}, \text{provider}, \text{type}, \text{region}, \text{status}, \text{metadata})
\]
used throughout the platform.

\subsection{Pseudo-code for Resource Canonicalization}
\begin{lstlisting}[language=Python, caption=Provider Resource Normalization]
def normalize_resource(provider: str, payload: dict) -> dict:
    if provider == "aws":
        return {
            "provider_id": payload.get("InstanceId") or payload.get("BucketName"),
            "name": extract_aws_name(payload),
            "provider": "AWS",
            "resource_type": infer_aws_type(payload),
            "region": payload.get("Region", "unknown"),
            "status": map_aws_state(payload),
            "metadata": payload,
        }
    if provider == "azure":
        return {
            "provider_id": payload.get("id"),
            "name": payload.get("name"),
            "provider": "Azure",
            "resource_type": infer_azure_type(payload),
            "region": payload.get("location", "unknown"),
            "status": map_azure_state(payload),
            "metadata": payload,
        }
    if provider == "gcp":
        return {
            "provider_id": payload.get("id") or payload.get("selfLink"),
            "name": payload.get("name"),
            "provider": "GCP",
            "resource_type": infer_gcp_type(payload),
            "region": extract_gcp_region(payload),
            "status": map_gcp_state(payload),
            "metadata": payload,
        }
    raise ValueError("Unsupported provider")
\end{lstlisting}

This normalization layer is strategically important because it creates a stable boundary for the rest of the platform. Once a resource has been canonicalized, the frontend does not need to know whether the original cloud payload contained fields such as \texttt{provisioningState}, \texttt{InstanceState}, or \texttt{status}. That difference becomes a backend concern.

\section{Task Scheduling and Retry Model}
Cloud execution is unreliable in the short term even when the requested action is valid. APIs throttle, locks occur, capacity becomes temporarily unavailable, and dependent services may take time to reach a ready state. Nebula addresses this through a bounded retry model.

Let a task $T_j$ have an initial submission time $t_0$ and retry counter $r_j$. For retryable failures, the next scheduled attempt is defined by exponential backoff:
\[
t_{j,r+1} = t_{j,r} + b \cdot 2^r + \epsilon
\]
where $b$ is the base delay and $\epsilon$ is a jitter term used to avoid synchronized retries across many tasks.

\section{Mathematical Model for Credential Security}
The security vault protects sensitive cloud secrets using a symmetric encryption model. We formalize this using standard cryptographic notation.

\subsection{Encryption Foundation}
Let $K$ be the unique, 128-bit master secret key stored as an environment variable.
Let $M$ be the plain-text cloud credential (e.g., Azure Client Secret).
Let $IV$ be a randomly generated Initial Vector (to prevent identical messages from having identical ciphertexts).

The encryption process $E$ can be defined as:
\[ C = E(K, M, IV) \]
where $C$ is the resulting ciphertext. In the Nebula vault, we use a symmetric construction exposed through the Fernet interface, which combines confidentiality and message authentication. The essential property needed by the platform is that the database stores only ciphertext and associated metadata, never the original secret.

If two identical credentials $M_1 = M_2$ are encrypted independently, randomized encryption ensures
\[
E(K, M_1, IV_1) \neq E(K, M_2, IV_2)
\]
whenever $IV_1 \neq IV_2$. This prevents direct equality inference from stored ciphertext values.

\subsection{Decryption at Runtime (Worker Scope)}
When a Celery worker picks up a task, it must materialize the credential $M$ to execute the Terraform plan. The decryption process $D$ is:
\[ M = D(K, C, IV) \]
The security property ensured here is that even if the database is compromised, the cloud accounts remain better protected because $K$ is never stored in the database. More precisely, the compromise of storage alone does not imply compromise of the original credential. The attacker must also obtain the application-layer key material from the runtime environment.

\subsection{Operational Security Assumptions}
The mathematical description is only meaningful when combined with operational assumptions. Nebula assumes:
\begin{enumerate}
    \item the key $K$ is injected through a secure runtime channel rather than hard-coded in source,
    \item worker memory is treated as trusted only for the duration of the task,
    \item execution logs never print decrypted secret values,
    \item key rotation invalidates old encryptions in a controlled migration path.
\end{enumerate}
These assumptions are what connect the notation to a secure implementation practice.

\section{Complexity Analysis of Resource Syncing}
Nebula must periodically sync every user's resource inventory across multiple clouds.
Let $N$ be the number of integrated cloud providers (AWS, Azure, GCP).
Let $R_i$ be the number of resources in the $i$-th cloud provider.
Let $T$ be the polling interval.

The total computational complexity of a full sync cycle is $O(\sum_{i=1}^{N} R_i)$. Since we use an asynchronous task queue, the perceived latency for the user is $O(1)$ relative to the sync operation itself.

If the sync work is parallelized across $W$ worker slots, the idealized processing time for a single cycle becomes
\[
\Theta \left( \frac{\sum_{i=1}^{N} R_i}{W} \right)
\]
subject to provider API rate limits and serialization points such as database writes. In practice, the observed time is bounded below by the slowest provider batch and bounded above by the total sequential scan.

\section{Decision Function for Cloud Selection}
Although the current platform allows the user to choose the target provider directly, the architecture also supports an advisory provider-selection model. For a candidate provider $p$, Nebula can compute a suitability score
\[
S(p) = w_c C(p) + w_l L(p) + w_r R(p) + w_s Sec(p) + w_a A(p)
\]
where:
\begin{itemize}
    \item $C(p)$ is the normalized cost score,
    \item $L(p)$ is the latency or proximity score,
    \item $R(p)$ is the regulatory fit score,
    \item $Sec(p)$ is the security-policy compatibility score,
    \item $A(p)$ is the availability and capacity score.
\end{itemize}
The weight vector can be tuned based on workload class. For a regulated workload, $w_r$ and $w_s$ may dominate. For an experimentation workload, $w_c$ and $L(p)$ may carry more influence. This formulation is useful because it turns future automation features into explicit optimization problems instead of ad hoc heuristics.

\section{Discussion}
The mathematical models in this chapter clarify the behavior of Nebula's core engine. The AI copilot is framed as a bounded classification-and-mapping problem, the resource inventory layer is expressed as a canonicalization function, retry behavior is described through a stateful backoff schedule, and credential protection is represented as controlled encryption and decryption over a secret key boundary. These formal views make the system easier to test, reason about, and extend. They also provide a strong bridge between the academic goals of the blackbook and the practical decisions embodied in the implementation chapters.
