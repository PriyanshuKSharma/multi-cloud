\chapter{SYSTEM DESIGN}

The system architecture of Nebula is designed to be highly modular, asynchronous, and secure. This chapter details the "Internal Workings" of the platform, focusing on the high-level ecosystem, database design, and the security model.

\section{System Architecture Overview}
The platform follows a three-tier architecture: Presentation, Application, and Persistence layers. Each layer is isolated using Docker-Compose to ensure a reproducible deployment environment.

\section{Network Architecture and Cloud Connectivity}
Provisioning resources in the cloud requires more than just API keys; it requires a structured networking foundation. Nebula manages this through VPC Peering and Security Group automation.

\subsection{Virtual Private Cloud (VPC) Design}
For every project launched in Nebula, a dedicated VPC is created. This ensures:
\begin{itemize}
    \item \textbf{Isolation}: Resources from one project cannot talk to another project's resources by default.
    \item \textbf{Secure Subnetting}: Public subnets for Load Balancers and private subnets for Database/Worker nodes.
    \item \textbf{Internet Gateway (IGW)}: For resources requiring external connectivity.
\end{itemize}

\section{Security Architecture Details}
Nebula implements a "Defense in Depth" strategy to protect user data and cloud access keys.

\subsection{Identity Layer (JWT Flow)}
User authentication is handled via JSON Web Tokens (JWT). When a user logs in:
1. The backend verifies credentials against PostgreSQL.
2. A signed JWT (RS256) is issued to the browser.
3. Every subsequent API call must include this token in the header as: \texttt{Authorization: Bearer <TOKEN>}.

\subsection{Credential Security (AES-128 Fernet)}
Cloud secrets (AWS Secret Keys) are never stored in plain text. We use the Fernet implementation which provides symmetric encryption with:
\begin{itemize}
    \item \textbf{Confidentiality}: Messages cannot be read without the key.
    \item \textbf{Integrity}: Messages cannot be altered without detection.
    \item \textbf{Freshness}: Messages cannot be replayed without detection.
\end{itemize}

\section{Component Interaction (TikZ Sequence)}
The interaction between the API, Redis, and workers is the core performance driver. When a user requests a sync:
1. API issues a \texttt{sync_all.delay()} call.
2. Redis stores the job.
3. The worker fetches the job, iterates through all registered cloud accounts, and updates PostgreSQL.
