\chapter{DEVOPS PIPELINE AND RELEASE ENGINEERING}
\thispagestyle{plain}

In addition to being a piece of software, Nebula also constitutes a way of deploying the software. As such, the surrounding DevOps pipeline falls within the technical scope of the project. The infrastructure manager that governs multiple clouds also needs to be deployed through a reliable, reproducible, and low-risk release cycle. This chapter discusses the release engineering principles behind that endeavor.

\section{Source Control and Branching Strategy}
Code management benefits greatly from a branching policy that distinguishes development and release activities. An efficient structure consists of:

\begin{itemize}
    \item \textbf{main}: branch for production releases with merge protection,
    \item \textbf{develop}: branch for integrating finished features,
    \item \textbf{feature/*}: task-specific branches,
    \item \textbf{hotfix/*}: emergency fixes for critical production problems.
\end{itemize}


The strategy minimizes risk during the release cycle by making it possible to review infrastructure changes, frontend changes, and backend changes simultaneously prior to deploying into a shared environment. Traceability is ensured since each release can be traced to its source code, pull requests, and approvals.

\section{Build Artifacts and Versioning}
Nebula creates several artifacts ready for deployment:

\begin{enumerate}
    \item frontend static resources,
    \item backend API service image,
    \item worker container image,
    \item Terraform module template and shared configuration bundle, 
    \items database migrations scripts.
    \item database migration scripts.
\end{enumerate}

All these elements need to be released under consistent versioning using a specific tag, such as \texttt{v1.4.0}. The important point here is to align the versions of the API service image, worker service image, and the database migration status against the same release. Otherwise, the debugging process will become unnecessarily complex because various elements may be based on conflicting assumptions.

\section{Continuous Integration Stages}
The CI process serves as the first filter for ensuring software quality. In its minimal implementation, the continuous integration tool should run:

\begin{itemize}
    \item backend static analysis and unit testing,
    \item frontend linting and build verification,
    \item syntax validation of Terraform templates,
    \item migration schema validation,
    \item API contract or smoke test.
\end{itemize}


The interesting aspect of this approach is that Terraform templates are considered as code artifacts. Although both the backend services and front end applications have passed all necessary tests, incorrectly structured Terraform template can make the entire platform unusable. Thus, the IaC layer must be tested along with the application layer.

It is critical that releases advance like this because it is possible for multi-cloud control plane releases to have issues that regular unit tests will not discover. This is because when deploying, problems may arise during the following scenarios:

\begin{enumerate}
    \item merge reviewed code into the integration branch,
    \item automatically deploy to a shared test environment,
    \item run smoke tests on login, queuing, and provider connection,
    \item deploy the same artifact to the staging environment,
    \item approving deployment to production,
    \item closely watch early metrics after deploying to production.
\end{enumerate}

This progression matters because a multi-cloud control plane can fail in subtle ways that ordinary unit tests do not catch. A release may look healthy until it attempts provider authentication, state reconciliation, or asynchronous task execution with live infrastructure APIs. Promotion through increasingly realistic environments is therefore a core risk-reduction strategy.

\section{Secrets and Configuration Management}
The release pipeline should know how to separate code from environment-dependent sensitive data. The following list contains items that should never be kept in source control:


The release pipeline must distinguish between code and environment-specific secret material. The following items should never be embedded in source control:
\begin{itemize}
    \item master encryption keys,
    \item production database credentials,
    \item service principal password,
    \item third-party API tokens.
\end{itemize}


These configuration parameters should be injected at runtime via environmen- tal variables or secret management systems. This will help with secure rotation, minimize risk of leakage and maintain portability of the codebase across environments.

\section{Containerization Strategy}
Containerization will bring value to the Nebula application since it will guaran- tee consistent execution environment. API, workers, databases, and brokers are composable components that can be reproduced by developers on their local environments. Benefits include:

\begin{itemize}
    \item consistent runtime dependencies,
    \item easy onboarding process,
    \item versioned artifact deployments,
    \item defined boundaries of services,
    \item rollback mechanism to known good versions.
\end{itemize}


Containerization is not enough to ensure operational reliability. Container images require additional review of vulnerabilities, health checks, startup probes, and proper logging. It is particularly critical for the worker container which uses secrets and manages the execution of Terraform.

\section{Database Migration Discipline}
Migration of schemas is another frequent cause of fragile releases. Nebula holds information about users, credentials, tasks, and resource states; an irresponsible migration could impact security and availability. In such a case, a well-thought-out migration strategy should consider:

\begin{enumerate}
    \item migrating forward only when possible,
    \item reviewing nullable-to-non-nullable changes explicitly,
    \item performing data backfills as release steps,
    \item backing up data before any destructive migration steps,
    \item conducting smoke tests on the migrated schema.
\end{enumerate}

\section{Rollback Strategy}
Each part of a release has its rollback strategies:



\begin{itemize}
    \item \textbf{API code rollback}: rdeploying previous versions of API and worker images. 
    \item \textbf{Configuration rollback}: reverting to proven configurations and feature flags.
    \item \textbf{Infrastructure rollback}: mplementing reverse changes from a thorough review, not assuming immediate cloud infrastructure rollback.
\end{itemize}


The difference between the three kinds of rollback is important since it might be that the release failed in one domain but succeeded in others. For instance, the API code is stable enough to be rolled back, whereas the database schema still supports backward compatibility.

\section{Discussion}
DevOps pipeline is no different from any other element that comprises the product itself; this element constitutes an integral part of the reliability envelope of the product. The idea behind Nebula is to decrease the complexity associated with operations for the sake of its users. For the same reason, the development team needs to practice discipline while building the very platform itself by applying such measures as versioning of artifacts, validation of infrastructure templates, use of secrets management outside the source code, and release promotion across multiple stages.
